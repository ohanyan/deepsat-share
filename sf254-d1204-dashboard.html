A one-read concept for our SF254-D1204 D2P2 submission. Enough to discuss scope, team shape, and evidence anchors — not a technical volume or a SOW. Decisions made here drive V2 drafting; open questions are flagged at the bottom for the team.

1. The Opportunity

The Space Development Agency (SDA) has published a D2P2 topic asking for a Secure Multi-Source Data Fusion Environment that lets proliferated LEO constellations (pLEO, PWSA/BMC3 Layer) ingest, fuse, and act on data from many vendors at once without leaking any single vendor's IP to the others.

The core problem the Space Force is trying to solve: BMC3 has to combine observations from a dozen different commercial and DoW satellite vendors to generate a decision. Today that fusion is brittle, vendor-dependent, and leaks proprietary tradecraft. SDA wants a vendor-neutral sandbox — an orchestration and fusion environment where any qualified provider can drop their sensor feed in, get fused against the others, and come back out as an actionable intel product for the operator, without the vendors seeing each other's algorithms.

The submission cycle is the DoD SBIR 25.4 BAA, Release 12, DAF Direct-to-Phase-II. $1.8M cap, 24-month period of performance, 25-page V2 technical volume.

This is not a satellite-hardware topic. It is a platform topic — which is exactly what DeepSat builds.

2. Why DeepSat, Why Now

DeepSat is not a satellite hardware company. DeepSat is an intelligence platform company — the constellation is one sensor of many feeding a broader orchestration layer. Our platform has four layers already under active construction:

  1. Orchestration platform and API — task any sensor, route any feed
  2. Edge AI models — trained on proprietary VLEO data, architected to run against any imagery source
  3. Sensor fusion engine — cross-cueing (not tip-and-cue) between space, aerial, and surface assets
  4. Payload abstraction — the interface layer that treats any satellite, sensor, or vendor as a plug-in

What SF254-D1204 asks for is, essentially, layers 1 + 3 + 4 hardened into a multi-vendor sandbox. We don't have to build a new company to pursue this topic; we have to harden, document, and certify what we're already building.

The framing that works across investor and customer conversations: "VLEO is to satellite intelligence what cloud was to enterprise software." This topic is the cloud control plane for the BMC3 Layer.

3. The Concept (what we'd build)

A Secure Multi-Source Data Fusion Environment — working name "Fusion Sandbox" for now, final product name TBD — with four functional surfaces:

A. Vendor Onboarding Surface

Any qualified satellite vendor (commercial or DoW) can register a sensor feed, declare its schema, and start pushing data into the sandbox. DeepSat's existing payload abstraction layer becomes the vendor-onboarding spec. This is the "how does a new vendor plug in" surface.

B. Secure Isolation Layer

Each vendor's raw data, algorithms, and model weights are isolated from the others. Fusion happens against outputs, not inputs — meaning two vendors never see each other's imagery or tradecraft, only each other's derived detections. This solves the proprietary-IP-leakage problem currently blocking real multi-vendor fusion at SDA.

C. Fusion and Decision Layer

Cross-cueing logic over the isolated outputs: if vendor A detects a vessel and vendor B's SAR pass confirms the same location, emit a fused high-confidence track. The operator (BMC3, SOF, or other) sees one fused picture, not twelve overlapping ones. This is the layer where DeepSat's edge AI and sensor fusion engine already live — we extend them to operate on multi-vendor inputs.

D. Operator Delivery Surface — TAK / ATAK

The fused outputs show up where the warfighter already lives: the Team Awareness Kit common operating picture. Whatever the satellite detected and whatever the AI fused about it only matters if it lands on the operator's ATAK screen as a standard Cursor-on-Target (CoT) event. We already have a scoped integration plan for this (see §5, Anchor C).

None of these four surfaces are new research. All of them exist in draft or production form inside DeepSat's current architecture. What the D2P2 funds is the hardening, certification (ITAR, zero-trust, FedRAMP-equivalent), multi-vendor onboarding, and BMC3-layer interop work that turns an internal platform into a deployable sandbox.

5. Phase I-Equivalent Evidence

Under the 25.4 BAA, D2P2 submissions must show Phase I-equivalent technical work already substantially performed by DeepSat (cannot rely on prior SBIR/STTR-funded work). We have three strong evidence anchors:

Anchor A — Active AFRL D2P2 "Orion's Belt" (FA864925P0298)

Already in execution. Documented performance: sub-100ms inference, >90% detection accuracy, <30s end-to-end latency from capture to decision, 100% encrypted data paths. A live government contract with AFRL demonstrating the edge-AI-on-space-imagery core that the Fusion Sandbox would extend. We cannot cite the D2P2-funded work itself as feasibility (rules disallow prior SBIR-funded work), but we can cite the architecture and capabilities that existed before and alongside that contract.

Anchor B — SOCOM Technical Exercise 26-2 (Avon Park, April 13-17, 2026)

Just completed last week. Over four days at Avon Park, DeepSat operated alongside SOCOM components (PEO TIS, PEO SDA, SOF AT&L S&T, SOFWERX) and demonstrated cross-cueing between satellite detections and aerial assets. Relevant takeaways:

  • Demonstrated cross-cueing (satellite ↔ aerial custody handoff) with MQ-9 and go-fast-vessel scenarios — the bi-directional fusion pattern at the heart of the Fusion Sandbox concept.
  • Confirmed five priority partnership pathways, one of which is Raytheon BBN / VPN overlay on TAC — the BMC3-layer delivery mechanism for our fused outputs.
  • Confirmed SOFWERX / PEO TIS / PEO SDA receptivity to DeepSat as an integrator, not just a sensor.
  • Secured verbal pathway for a CRADA (no matching funds required) and potential C-130 / MQ-9 flight access for sensor validation.

Freshest, most vivid Phase I-equivalent evidence, and all unclassified commercial exercise work.

Anchor C — TAK Server / ATAK Integration Architecture

Full integration architecture already scoped (see notion-export/…/ATAK Team Awareness Kit.md, Craig, March 2026): CoT gateway converting DeepSat detections into MIL-STD-2525 symbology, confidence-filtered and AOI-bounded delivery to the operator's ATAK common operating picture, mTLS connection to customer-hosted TAK Server. 2–3 weeks of engineering for the MVP gateway; industry reference patterns include SkyFi's full plugin and BAH's Sit(x) bridge adapters. Not speculative — it's a scoped engineering plan with library choices, protocol decisions, and identified test harness (FreeTAKServer + ATAK-CIV on Android).

Use in V5 Feasibility Documentation. Anchor A establishes we've operated a real USG contract at scale. Anchor B establishes that the cross-cueing and multi-vendor fusion concept has already been operationally exercised alongside SOF customers. Anchor C establishes an engineered path to getting fused outputs onto the warfighter's screen.

6. Suggested Project Structure (24-month PoP, concept level)

A clean way to slice 24 months at concept level — not yet a SOW, not yet a budget, just something the team can argue with. Each phase ends with a government-visible demonstration; no long periods of invisible development.

Phase 1 (Months 1-6) — Vendor Onboarding Surface + Secure Isolation Spec

  • Harden payload abstraction layer into a documented multi-vendor onboarding contract
  • Define secure-isolation model (outputs-only fusion, no raw data or weights shared between vendors)
  • First vendor other than DeepSat onboarded as reference integration — candidate: Raytheon-adjacent pLEO feed, or a SOCOM-sourced commercial partner

Deliverable: multi-vendor onboarding spec + reference implementation with 2 vendors.

Phase 2 (Months 6-12) — Fusion and Decision Layer (cross-cueing at scale)

  • Extend fusion engine from "space + aerial" (current) to "multi-vendor space + aerial + surface"
  • Cross-cueing logic on isolated vendor outputs, not inputs
  • Confidence scoring and conflict resolution when vendors disagree

Deliverable: fusion engine operating against 3+ isolated vendor feeds in a controlled test environment.

Phase 3 (Months 12-18) — Operator Delivery (TAK / ATAK / BMC3)

  • Ship the CoT gateway and TAK Server integration (Anchor C, productionized)
  • Raytheon BBN VPN overlay / TAC integration for the BMC3 layer
  • Operator usability work with a named SOF or DAF end user on the receiving side

Deliverable: fused outputs visible on an operator's ATAK COP in a government exercise.

Phase 4 (Months 18-24) — Hardening, Certification, Transition

  • ITAR, zero-trust, FedRAMP-equivalent certification
  • Transition plan to a production BMC3 or SOF-adjacent program of record
  • Phase III candidates: PEO TIS, PEO SDA (SOCOM), SDA directly, or a named DAF PoR

Deliverable: sandbox deployable into a real operational network with signed-off security documentation.
